# cyber-emperor

## 适用目标

当任务满足以下任一条件时启用本 skill：

- 需要把一个中大型任务拆成多个明确子任务并行推进
- 需要多个 Agent 分工：分析、实现、测试、文档、验收
- 需要跨文件、跨模块、跨阶段协作
- 需要主会话负责编排，而具体产出交给分身执行
- 任务中存在 **复杂编码**（>50 行代码、多文件改动、重构、新功能开发）

一句话：**当任务已经不是“直接做完”，而是“需要一套协作编排机制”时，用 cyber-emperor。**

---

## 不适用场景

以下情况不要用：

- 单文件、小改动、十分钟内可直接完成的任务
- 纯查询类任务：读文件、搜信息、查日志、回答概念问题
- 明显必须串行且拆分后会频繁冲突的任务
- 需要持续互动、边聊边改方向的任务
- 只改 1~20 行代码、且无需多人协作的简单修补

一句话：**不要为了“显得高级”而强行上多 Agent。拆分成本高于收益时，不用。**

---

## 核心原则

1. **本体只统筹，不承担主要编码产出**
2. **所有实际子任务通过 `sessions_spawn` 下发给分身**
3. **复杂编码任务必须由分身结合 `claude-code-hook` 执行**
4. **分身之间避免同时修改同一文件，优先按模块边界拆分**
5. **每个分身必须有明确输入、输出、验收标准**
6. **主会话负责最终合并、验收、汇报与回收分身**

这是一个工程编排 skill，不是叙事设定。重点是：**谁做、做什么、何时验收、如何收口。**

---

## 三省六部映射

将多 Agent 协作抽象为“三省六部”结构，方便稳定复用。

### 三省

#### 1. 中书省（任务设计）
负责把用户目标翻译为可执行计划：

- 明确范围、约束、交付物
- 判断是否适合多 Agent
- 进行任务拆分
- 定义每个分身的职责与验收标准

通常由 **主会话本体** 承担。

#### 2. 门下省（审查校验）
负责在执行前后做审查：

- 检查拆分是否合理
- 检查任务间是否会冲突
- 审核分身结果是否满足目标
- 必要时打回重做

可由本体承担，也可单独放一个“审查分身”。

#### 3. 尚书省（执行调度）
负责实际派工与回收：

- 调用 `sessions_spawn`
- 跟踪各分身职责
- 接收结果
- 汇总验收
- 调用 `subagents kill` 回收分身

通常由 **主会话本体** 承担。

### 六部

六部不是固定要六个分身，而是六类职责槽位。按需启用。

#### 吏部：角色分配
- 定义分身 label
- 规定每个分身负责的模块/目录/阶段
- 避免职责重叠

#### 户部：资源与边界
- 记录项目路径、技术栈、依赖命令
- 明确哪些文件可改、哪些不能碰
- 明确时间预算和产出格式

#### 礼部：接口与规范
- 统一代码风格、目录规范、提交格式、文档格式
- 统一返回内容：改了什么、怎么验收、有什么风险

#### 兵部：执行推进
- 负责真正派出分身干活
- 对复杂编码任务强制挂接 `claude-code-hook`

#### 刑部：质量与纠错
- 跑测试、lint、type-check、build
- 审查逻辑缺口、边界情况、回归风险
- 失败则回退到对应执行分身重做

#### 工部：交付与文档
- 汇总变更
- 维护 README / ARCHITECTURE / 示例文档
- 形成最终可交付结果

## 标准工作流

## 第一步：判断是否启用 cyber-emperor

优先问自己四个问题：

1. 任务是否需要 2 个以上角色协作？
2. 是否存在明显可并行的模块边界？
3. 是否有复杂编码必须交给 Claude Code？
4. 本体是否更适合做编排而不是亲自下场？

若大多为“是”，启用本 skill。

---

## 第二步：按三省六部拆分

推荐拆成以下几类子任务：

- 方案/拆分分身
- 实现分身（一个或多个）
- 测试/验收分身
- 文档/收口分身

典型分配示例：

- 中书省：本体做任务拆分
- 门下省：验收分身负责审查
- 尚书省：本体调度分身
- 兵部：实现分身 A / B
- 刑部：测试分身
- 工部：文档分身

---

## 第三步：强制使用 sessions_spawn

只要启用本 skill，**实际执行必须由分身承担**。本体不能口头说“我来模拟多 Agent”，而不真正派工。

分身任务必须至少包含：

1. 角色定位
2. 项目路径
3. 先读哪些文件
4. 具体任务清单
5. 输出要求
6. 验收命令或验收标准

### 分身任务模板

```text
你是“大拿-XXX分身”，负责 [具体职责]。

项目路径：/path/to/project
请先阅读：
1. /path/to/AGENTS.md
2. /path/to/相关文档...
3. ~/.nvm/versions/node/v24.13.0/lib/node_modules/openclaw/skills/claude-code-hook/SKILL.md （若涉及复杂编码）

任务要求：
1. ...
2. ...
3. ...

输出要求：
- 说明改了哪些文件
- 说明核心实现/结论
- 说明如何验收

验收标准：
- [命令或条件]
```

---

## 第四步：复杂编码强制挂接 claude-code-hook

只要某个分身任务符合以下任一条件，就必须在分身任务里明确要求使用 `claude-code-hook`：

- 预计新增或修改代码 > 50 行
- 涉及 3 个以上文件
- 新功能开发
- 重构、迁移、架构整理
- 需要较长时间独立完成的编码任务

### 强制规则

- **不能让分身直接裸写大段代码**
- **不能用轮询方式傻等 Claude Code**
- **必须按 `claude-code-hook` 的零轮询模式执行**

### 推荐嵌入到分身任务中的约束段

```text
此任务属于复杂编码，强制结合 claude-code-hook：
1. 先读取 claude-code-hook 的 SKILL.md
2. 用 Claude Code 异步执行，不要轮询刷状态
3. 等待 Hook 回调后再做验收
4. 完成后汇报：修改文件、核心实现、验收结果、风险项
```

---

## 第五步：执行中的组织纪律

### 本体职责

- 只做拆分、派工、汇总、验收、回收
- 不直接承担主要代码编写
- 控制并发数量，避免文件冲突
- 在分身完成后及时汇总结果

### 分身职责

- 只完成自己负责的边界内任务
- 不擅自扩需求
- 不改不属于自己职责的区域
- 复杂编码按 hook 流程执行

### 冲突规避

优先按以下边界拆：

- 按模块拆：`src/auth` / `src/ui` / `docs/`
- 按阶段拆：实现 / 测试 / 文档
- 按产物拆：代码 / 配置 / 说明文档

避免：

- 两个分身同时改同一个核心文件
- 一个分身边写代码，一个分身同时重构同目录
- 未定义验收标准就直接开工

---

## 第六步：门下省审查与刑部验收

至少做以下一种验收：

- `pnpm lint`
- `pnpm type-check`
- `pnpm test`
- `pnpm build`
- 文档完整性检查
- 关键路径人工审查

如果是文档类 skill / 流程类产出，验收至少包括：

- 结构是否完整
- 是否说明“什么时候用 / 不用”
- 是否给出可直接复用的模板
- 是否写明依赖 `sessions_spawn` 与 `claude-code-hook`

门下省的核心职责不是重复劳动，而是：**判断结果能不能交付。**

---

## 第七步：尚书省收口与回收

分身完成后，本体要做三件事：

1. 汇总各分身结果
2. 做最终验收
3. 及时 `subagents kill` 回收分身

不要让分身长期驻留。

---

## 最小执行模板

适用于大多数工程任务。

### 模板 A：两分身

- 分身 1（兵部）：实现
- 分身 2（刑部/工部）：测试 + 文档

适合：
- 中等复杂任务
- 一个主功能 + 一个验收收口

### 模板 B：三分身

- 分身 1（兵部甲）：模块 A
- 分身 2（兵部乙）：模块 B
- 分身 3（刑部/工部）：测试、验收、文档

适合：
- 两块可并行开发的模块
- 最后需要统一验收

### 模板 C：四分身

- 分身 1：方案细化 / 风险预判
- 分身 2：实现 A
- 分身 3：实现 B
- 分身 4：测试与文档

适合：
- 较大任务
- 需要前置分析、并行实现、独立验收

```text
使用 cyber-emperor 模式处理此任务。

按三省六部进行：
- 中书省：先完成任务拆分与职责定义
- 尚书省：使用 sessions_spawn 派出分身执行
- 门下省：最后审查结果是否可交付
- 兵部：负责实际实现
- 刑部：负责测试与纠错
- 工部：负责文档与交付收口

硬性要求：
1. 实际执行必须使用 sessions_spawn
2. 凡是复杂编码分身，必须先读 claude-code-hook 并按零轮询模式执行
3. 不要让两个分身同时改同一文件
4. 每个分身都要有明确输入、输出、验收标准
5. 最终汇总时给出：改动文件、核心结果、验收方式、剩余风险
```

## 分身 task 示例（复杂编码）

```text
你是“大拿-认证模块兵部分身”，负责实现认证模块改造。

项目路径：/Users/example/project
请先阅读：
1. /Users/example/project/AGENTS.md
2. /Users/example/project/ARCHITECTURE.md
3. ~/.nvm/versions/node/v24.13.0/lib/node_modules/openclaw/skills/claude-code-hook/SKILL.md

任务要求：
1. 梳理现有认证流程
2. 完成 auth service 重构
3. 补齐类型与错误处理
4. 不修改与认证无关的 UI 文件

强制要求：
- 此任务属于复杂编码，必须结合 claude-code-hook
- 不要轮询等待，使用 Hook 回调机制

输出要求：
- 列出修改的文件
- 说明核心实现
- 给出验收命令与结果
- 说明遗留风险

验收标准：
- pnpm type-check
- pnpm test src/auth
```

## 示例调用场景

下面给出几种适合直接套用的场景，让主会话能快速判断阵型与派工方式。

### 场景 1：新功能完整交付

任务特征：
- 需要需求澄清、方案设计、编码实现、测试、文档
- 通常会跨多个目录或多个阶段

推荐阵型：标准三分身或四分身
- 中书省：本体拆解需求与成功标准
- 兵部分身 A：实现核心功能
- 兵部分身 B：实现配套模块或 UI / API 对接
- 刑部 / 工部分身：测试、文档、交付说明

启用要点：
- 两个实现分身不能改同一核心文件
- 只要实现超过 50 行或涉及多个文件，强制接入 claude-code-hook
- 最终由本体统一验收并汇总交付

### 场景 2：多模块重构

任务特征：
- 需要先盘点影响面
- 存在多个可拆模块，但也可能有公共文件冲突风险

推荐阵型：四分身
- 分身 1：影响面分析 / 风险预判
- 分身 2：模块 A 重构
- 分身 3：模块 B 重构
- 分身 4：类型检查、构建、回归验证、文档更新

启用要点：
- 先分析再动手，避免上来就并行改炸
- 公共底层文件若冲突风险高，应改为串行
- 重构任务默认视为复杂编码，优先接入 claude-code-hook

### 场景 3：研究 + 方案 + 交付建议

任务特征：
- 不一定大量写代码，但需要多角色协同
- 需要搜集、比较、归纳、形成可执行建议

推荐阵型：两分身或三分身
- 分身 1：资料搜集与事实整理
- 分身 2：方案对比与推荐结论
- 分身 3（可选）：整理成用户可直接阅读的交付文档

启用要点：
- 重点不在 Hook，而在角色边界和输出结构统一
- 礼部职责要更重，确保最终材料不是拼盘

### 场景 4：文档体系整理与工程交付收口

任务特征：
- 代码已基本完成，但 README、ARCHITECTURE、示例、变更说明未完善
- 需要把零散成果整理成可交付状态

推荐阵型：两分身
- 分身 1：扫描现有改动并补全技术文档
- 分身 2：验收文档一致性、命令可执行性、目录引用正确性

启用要点：
- 此类场景不一定需要 claude-code-hook
- 更关注礼部（统一表达）与工部（交付文档）协作

## 对主会话的硬约束

启用 cyber-emperor 后，主会话应避免：

- 自己直接写完整实现代码
- 一边说多 Agent，一边没有真正 spawn 分身
- 复杂编码任务不用 claude-code-hook
- 没有验收就直接宣布完成
- 不回收分身

如果做了以上任一项，说明这个 skill 没有被正确执行。

---

## 成功标志

一个合格的 cyber-emperor 执行结果，至少应满足：

- 任务被明确拆分为多个职责单元
- 分身通过 `sessions_spawn` 实际派出
- 复杂编码分身明确结合 `claude-code-hook`
- 有独立验收动作，不是只看分身口头汇报
- 最终输出可交付，而不是停留在“计划很好”

---

## 一句话总结

`cyber-emperor` 不是用来“装阵仗”的，而是把复杂任务稳定落成：

- **中书省** 负责拆分
- **尚书省** 负责调度
- **门下省** 负责审查
- **六部** 负责分工执行
- **sessions_spawn** 是派工底座
- **claude-code-hook** 是复杂编码底座

没有这两者，就不算真正启用本 skill。
