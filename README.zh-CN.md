# 赛博皇帝（Cyber Emperor）

> **不是多开几个 Agent，是真正建立一套能打硬仗的 AI 朝廷。**

`cyber-emperor` 是一个为 OpenClaw 设计的高阶多 Agent 编排 skill。它把传统“三省六部”的治理智慧，翻译成现代 AI 协作系统里的 **任务拆解、分权协同、并行执行、风险审查、统一验收、最终交付**。

如果说普通多 Agent 只是“多找几个人一起上”，那么 `cyber-emperor` 做的是：

- 先定战略，再派兵
- 先划边界，再并行
- 先设验收，再开工
- 先管秩序，再谈效率

一句话：

**它不是让 Agent 变多，而是让复杂任务第一次真正拥有“组织能力”。**

---

## 它到底厉害在哪？

绝大多数多 Agent 系统，最大的问题不是 Agent 不够聪明，而是：

- 谁负责什么，说不清
- 多个 Agent 一起干，边界混乱
- 并行推进时互相踩文件、踩结论
- 输出看起来热闹，实际不可审计、不可复用、不可交付
- 本体一边说“分工协作”，一边自己又下场补锅

`cyber-emperor` 的目标，就是解决这些真实工程问题。

它不是空泛世界观，不是 cosplay，也不是中二包装壳。它解决的是复杂任务落地时最核心的五件事：

1. **怎么拆** —— 把大任务拆成真正可以执行的子任务
2. **谁来做** —— 每个角色边界清晰，不抢活不越权
3. **怎么并行** —— 能并行的并行，不能并行的绝不乱上
4. **怎么验收** —— 不是做完就算，而是有明确质量闸门
5. **怎么交付** —— 最终产物不是拼盘，而是一份能直接给人看的成品

---

## 核心定位：这不是“影分身加强版”，而是治理框架

很多人看到它，会以为这只是“更高级的多开分身”。

不，定位完全不同：

### `shadow-clone`
解决的是：
- **怎么并行放分身去干活**

### `claude-code-hook`
解决的是：
- **复杂编码如何零轮询、异步执行、低成本完成**

### `cyber-emperor`
解决的是：
- **这个任务为什么要拆？**
- **拆成哪些角色最合理？**
- **哪些环节该并行，哪些必须审批？**
- **复杂编码应该何时接入 Hook？**
- **做完之后谁来验收、谁来交付、谁来收口？**

所以关系非常清楚：

- **赛博皇帝**：负责定朝纲、控全局、做治理
- **影分身**：负责布阵、派兵、并行执行
- **Claude Code Hook**：负责工部重器级复杂建设

这三者一结合，才是一套真正能打项目战、交付战、复杂协同战的多 Agent 工作流。

## 三省六部：把古代治理智慧，翻译成现代 AI 协作架构

`cyber-emperor` 不是简单借个“皇帝”噱头，而是把“三省六部”的组织逻辑映射成现代多 Agent 协作系统。

### 三省：治理中枢

#### 1. 中书省 —— 战略起草与任务拆解
负责：
- 解析用户真实目标
- 拆出可执行子任务
- 规划依赖关系
- 设定完成标准
- 决定整体推进路线

这是任务从“模糊需求”走向“可执行计划”的第一步。

#### 2. 门下省 —— 风险审查与质量把关
负责：
- 复核方案是否合理
- 检查是否有文件冲突、角色冲突、流程冲突
- 挑出漏项、风险项、不可交付项
- 决定哪些动作需要人工确认

这一步决定系统不是“看起来很忙”，而是真的可靠。

#### 3. 尚书省 —— 调度、汇总、收口
负责：
- 派发分身
- 跟踪职责
- 汇总阶段成果
- 组织最终验收
- 做最终交付与回收

它是整个体系的执行中枢和交付总控。

---

### 六部：专业执行机器

#### 吏部 —— 角色与能力路由
决定：
- 哪个子任务交给谁
- 用普通分身、影分身，还是 Claude Code
- 分身数量如何控制，避免无脑扩编

#### 户部 —— 范围、资源、边界控制
决定：
- 哪些文件能改
- 哪些区域不能碰
- 时间预算、上下文预算、成本预算
- 哪些需求属于“暂不处理”

#### 礼部 —— 规范、表达、统一口径
负责：
- 输出格式统一
- 文档结构统一
- 结果表述统一
- 最终交付的阅读体验统一

#### 兵部 —— 并行执行与阵型调度
负责：
- 调用 `shadow-clone`
- 布置多分身阵型
- 处理并行推进
- 失败后重组编队

#### 刑部 —— 验收、追责、风控
负责：
- type-check / test / build / lint
- 风险排查
- 输出可否交付判断
- 不过线就打回重做

#### 工部 —— 真正建设产物
负责：
- 写代码
- 改结构
- 搭系统
- 补文档
- 产出真正可交付的成果

复杂编码场景下，工部优先联动 `claude-code-hook`，实现零轮询、低干扰、高质量建设。

## 它适合什么任务？

这个 skill 不是拿来处理鸡毛蒜皮小事的。它适合的是：

### 1. 项目级任务
例如：
- 新功能完整交付
- 一个新模块从 0 到 1
- 产品级功能拆分与落地

### 2. 多模块并行改造
例如：
- 前后端联动改造
- 多文件重构
- 多目录结构整理
- 多角色同步推进

### 3. 复杂工作流编排
例如：
- 需求分析 → 方案设计 → 实现 → 测试 → 文档 → 发布准备
- 技术调研 → 路线对比 → 建议输出 → 实施方案

### 4. 需要统一验收与统一交付的任务
例如：
- 文档体系整理
- 架构说明归档
- 项目对外交付版本整理
- 多 Agent 协作后的成果统一收口

### 5. 复杂编码任务
例如：
- 超过 50 行代码的功能开发
- 多文件重构
- 新组件 / 新模块 / 新服务创建
- 长时间独立编码任务

## 它不适合什么？

越厉害的工具，越要知道什么时候别乱用。

以下情况不要启用 `cyber-emperor`：

- 单文件小修
- 简单问答
- 查一个概念
- 看一个日志
- 十分钟内就能直接做完的任务
- 拆分收益明显低于沟通成本的任务

说白了：

**不要为了显得高级，就把本来一把小刀能解决的事，硬升级成出动满朝文武。**

## 这套 skill 的核心价值

### 1. 让复杂任务“可治理”
不是靠感觉做，而是靠结构做。

### 2. 让多 Agent 协作“可控”
不是大家一起上，而是每个人知道自己该干什么。

### 3. 让并行推进“可交付”
不是同时做很多事，而是同时做很多**不会互相撞车**的事。

### 4. 让复杂编码“可持续”
不是本体自己硬写，而是让专业执行器去做，降低 Token 浪费，提升质量。

### 5. 让最终结果“像一个团队做出来的”
不是碎片化产出，而是统一口径、统一结构、统一交付感。
